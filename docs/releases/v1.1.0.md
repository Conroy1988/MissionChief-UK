# MissionChief UK v1.1.0

**Released:** 23 July 2026  
**Release type:** Mission intelligence expansion  
**Static Data API:** v1, backwards compatible

MissionChief UK v1.1.0 expands Mission Lookup from the original 62 canonical mission records to the complete public United Kingdom mission catalogue, begins the enforced programme to reach 100% fully canonical coverage and preserves a strict distinction between official source data and normalized operational evidence.

## Mission catalogue baseline

```text
1,062 official UK mission records
79 canonical mission records
62 official IDs matched to canonical records
21 fully canonical mission records
1,000 official records awaiting direct canonical records
17 canonical overlay or derived records without standalone official IDs
46 canonical deployable-resource records
18 canonical infrastructure records
11 qualification records
Static Data API v1.1.0
```

The official source snapshot was retrieved from the public MissionChief UK mission feed and stored with SHA-256 provenance. Every official top-level and nested mission field is retained.

## Added

- A complete official UK mission catalogue containing 1,062 current mission records.
- Search across official mission IDs, names, POIs, generators, categories, requirement keys and prerequisites.
- Mission Lookup filters for service or generator and evidence coverage.
- Separate **Canonical mapped** and **Official UK catalogue** evidence badges.
- Structured expandable views for patients, personnel, duration, follow-up missions, expansion missions, overlays, variants and all published additional fields.
- A complete official JSON record viewer for every matched or official-only mission.
- Public read-only catalogue, reconciliation and verification endpoints under `assets/data/official/`.
- Source-to-canonical coverage reporting and official-key inventories.
- A deterministic offline integrity audit for the complete official catalogue.
- Browser acceptance coverage for catalogue completeness, reconciliation, official-only search, canonical search, full-record display and horizontal containment.
- A content-addressed daily refresh workflow with automatic Pages deployment only when source or derived verification state changes.

## 100% verification programme

Every official mission now progresses through five explicit gates:

1. captured;
2. identity verified;
3. requirements mapped;
4. operationally verified; and
5. fully canonical.

The release includes:

- a machine-readable mission-verification registry;
- an official-key mapping registry;
- deterministic per-mission blockers and next actions;
- strict key-equivalence validation for promoted missions;
- offline regeneration of official/canonical coverage after every canonical batch;
- a generated public verification endpoint; and
- a generated verification-status page exposing the complete backlog.

The first two fully canonical batches contain IDs:

```text
0, 1, 2, 3, 4, 6, 7, 8, 9, 10, 11,
13, 14, 15, 16, 17, 18, 19, 23, 24, 27
```

These 21 records cover low-complexity Fire and Rescue missions. Their official Fire Station prerequisite and Fire engine requirement keys are mapped explicitly, quantities are checked against canonical records, Forest and Bus stop POIs are retained, and published rewards, expansion paths and follow-up relationships remain structured.

## Accuracy controls

- Official source values are preserved verbatim.
- Unknown internal requirement keys are displayed as official keys rather than guessed into vehicle names.
- Canonical records remain the higher-trust dispatch model.
- Seasonal availability, base missions, overlays, additive overlays and variants remain distinct.
- New official fields are retained automatically instead of being silently dropped by a fixed allow-list.
- The public browser catalogue must preserve every field from the retained source snapshot.
- Duplicate IDs, missing names, stale reconciliation, stale key inventories and source/public mismatches fail validation.
- A mission cannot enter a mapped verification gate while any official requirement, chance or prerequisite key is unclassified.
- A `not-applicable` key classification must define a narrow allowed-value set; unexpected values fail validation.
- Strictly audited missions must exactly match official mapped quantities and canonical preconditions.

## Mission Lookup behaviour

Mission Lookup now loads both collections:

```text
assets/data/v1/missions.json
assets/data/official/uk-missions.json
```

The official catalogue is fetched once per page and shared by lookup, structured metadata and complete-record views. Results are limited to the first 100 matching cards for rendering performance, while all 1,062 records remain searchable.

## Public data additions

New endpoints:

```text
assets/data/official/uk-missions.json
assets/data/official/uk-mission-coverage.json
assets/data/official/uk-mission-verification.json
```

These endpoints are separate from the canonical `assets/data/v1/` contract. Existing v1 paths, collection envelopes, canonical IDs and evidence semantics remain compatible.

## Automated maintenance

The official catalogue refresh runs daily and may be dispatched manually. An unchanged source SHA and unchanged verification state produce no repository change. A changed source must pass importer validation, lossless publication, compaction, offline coverage reconciliation, official-key mapping validation and the complete catalogue audit before it can be committed and deployed.
