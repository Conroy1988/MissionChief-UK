# Verification Workflow

Stage 9 establishes a controlled path from community observation to published operational intelligence.

## Evidence states

| State | Meaning |
|---|---|
| `verified` | Reproduced in the current UK game or confirmed through a reliable primary source |
| `calculated` | Derived from verified values with the method documented |
| `community-tested` | Reproduced independently but awaiting full editorial verification |
| `review-required` | Potentially outdated, incomplete or not yet reproduced |
| `obsolete` | Confirmed no longer applicable to the current UK game |

## Submission requirements

A useful evidence submission includes:

- the exact UK-game context;
- what was observed;
- the date of observation;
- the relevant building, mission, vehicle or setting;
- screenshots or other evidence where appropriate;
- steps another contributor can follow;
- any conflicting result already known.

## Editorial process

1. **Intake** — classify the claim and identify the affected records.
2. **Reproduction** — repeat the observation where practical.
3. **Reconciliation** — compare with current documentation and structured data.
4. **Publication** — update all affected representations together.
5. **Traceability** — record the source and verification date.
6. **Review** — revisit time-sensitive records after relevant game updates.

## Conflict handling

Conflicting evidence must remain visible until resolved. The project should document differences in account configuration, mission variants, region, timing or interface state rather than selecting a preferred value without analysis.

## Attribution

Contributors should be credited for meaningful original evidence while personal or account-sensitive information is removed from published material.
